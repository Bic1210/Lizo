# Lizo 产品路线图

> 作者：Alex（Product Manager Agent）
> 日期：2026-04-20
> 版本：v1.0
>
> 本文件是 Lizo 上线后的产品决策参考。单人团队，每个优先级选择都是排他的——做了这个就没有资源做那个。请在执行前重读「核心假设」部分。

---

## 0. 核心假设

1. **团队规模**：1 名全栈开发者，约 60% 时间在产品上（其余是学业/其他）
2. **资金**：无外部融资，靠硬件销售维持 API 成本
3. **用户画像**：18–25 岁，中国大陆，夜晚孤独，ACG 亲和，微信/微博/小红书活跃
4. **竞品压力**：Character.ai 在国内被墙，国内同类产品情感深度不足，窗口期存在但不永久

---

## 1. 产品北极星指标

### 核心指标：**7日消息留存率**（7-Day Message Retention）

定义：新用户首次发消息后，7天内至少再发 1 条消息的比例。

**为什么是这个，而不是其他指标？**

| 候选指标 | 排除原因 |
|---------|---------|
| DAU | 无账号体系，无法追踪，MVP 阶段测量成本过高 |
| 页面 PV/UV | 流量虚荣指标，反映不了情感粘性 |
| 硬件销量 | 软件留存是硬件购买的前置条件，倒因为果 |
| 好评率/评分 | 主观且难收集，容易被刷 |
| 聊天消息总量 | 单次轰炸式聊天会虚高，掩盖真实留存 |

**为什么 7 日留存是 Lizo 的核心？**

Lizo 的价值主张是「陪伴」，陪伴的本质是「你明天还会回来」。一次性体验不是陪伴，是娱乐。7 日留存率衡量的是：用户是否把 Lizo 纳入了日常情绪出口。这个指标一旦健康（>30%），硬件购买意愿、口碑传播、付费转化都会自然跟随。

**MVP 基线目标**：上线第一个月，7 日消息留存 > 25%。

> 注意：现在无账号体系，7 日留存只能通过 localStorage 或用户自述收集，精度有限，但趋势有参考价值。真正精确追踪需要 v1.2 的账号体系。

---

## 2. v1.1 — 上线后第 1–2 周

### 主题：「让用户第二天还想回来」

MVP 最致命的问题不是 Bug，是**无记忆**。用户昨天和 Lizo 说的话，今天刷新全没了。情感产品的用户会把这种体验读解为「Lizo 不记得我」，这比任何 UI 问题都更伤害留存。

### 本阶段约束

- 后端基础设施不动（Flask 跑在 Pi，数据库 SQLite）
- 不引入用户注册/登录
- 开发时间预算：约 1.5 周

### 要做的 3 件事（按优先级）

**P0 — 聊天历史本地持久化**

在前端用 `localStorage` 存储最近 50 条对话记录。用户刷新或关闭后，下次打开还能看到历史。成本极低（纯前端，不改后端），但对用户感知的「Lizo 记得我」影响巨大。

实现细节：
- `Chat.tsx` 的 `messages` state 初始值从 `localStorage.getItem('lizo_chat_history')` 读取
- 每次 messages 更新时 `localStorage.setItem` 同步写入
- 限制 50 条防止 storage 溢出，超出时删最旧的

预期效果：用户第二天打开看到昨晚的对话，情感连续性建立。

---

**P1 — Night Mode 实装**

设计稿中 Night Mode 已完整定义（见 `docs/DESIGN.md` 双模式系统），代码里没有实装。这是 Lizo 最能拉开差距的感知层功能：夜晚孤独的用户打开是深蓝黑背景 + 紫色发光，氛围感直接翻倍。

实现路径：
- `index.css` 中的 Night Mode CSS 变量已有定义，需要用 `data-theme="night"` 属性切换 `:root`
- `App.tsx` 加入时间检测逻辑：20:00–07:00 自动切换 Night Mode，同时提供手动开关
- 手动偏好存入 `localStorage`（`lizo_theme_preference`）

优先夜间切换，因为目标用户（夜猫子）在这段时间最活跃。

---

**P2 — About 页填实内容（M-1 修复）**

审查报告里 M-1 已指出 About 页是空壳，两个 CTA 按钮都导向这里。上线后新用户点「了解硬件版本」看到「施工中」会直接关闭标签页。

最低限度做法（不需要 1 周）：
- 一张 Lizo 形象大图 + 一段 200 字介绍文案
- 硬件规格卡片（树莓派 + 伺服马达 + LCD 面板 + 触感传感器），附一张实物照片
- 「预约联系」按钮指向邮件或微信（暂时够用，不需要支付）

---

### v1.1 不做什么

- 用户注册登录（工程量大，优先验证留存）
- 社区功能（用户基数太小，社区是空的）
- 视频内容生产（即梦工具需要单独时间，不放在冲刺里）
- 移动端 iOS 键盘修复（N-1 Minor 问题，不阻断核心体验）

## 3. v1.2 — 第 3–4 周

### 主题：「认识你，而不只是聊天」

经过前两周上线，预期会收到的用户反馈：

| 反馈类型 | 预期来源 | 概率 |
|---------|---------|------|
| 「换设备后记录全没了」 | 用了 localStorage 的用户 | 高 |
| 「Lizo 不记得我之前说过的话」| 高频聊天用户 | 高 |
| 「可以看看 Lizo 今天的心情吗？」 | Soul 页用户 | 中 |
| 「我想分享给朋友」 | 有传播意愿的用户 | 中 |
| 「视频什么时候有？」 | 首页 VideoShowcase 看到占位符的用户 | 高 |

### 本阶段核心：轻量账号体系

这是 v1.2 最重要的基础设施工作。没有账号，7 日留存无法精确追踪，Lizo 的「记忆」无法跨设备，后续所有功能都是沙上建楼。

**方案：手机号 + 验证码（不做密码）**

- 理由：国内用户接受度高，开发简单，无密码遗忘问题
- 前端：登录弹窗，发送验证码，填写后登录
- 后端：SQLite 加 `users` 表（`id`, `phone_hash`, `nickname`, `created_at`），接入短信服务（阿里云短信，月费低）
- 会话：JWT token，存 localStorage，7 天有效

账号体系一旦建立，同步完成：
- 聊天历史迁移到后端（SQLite `messages` 表），多设备可用
- Soul 页数据与用户绑定，情绪曲线连续
- 7 日留存率开始可精确追踪

---

**第二优先：视频内容生产（业务侧，不是开发）**

这不是代码任务，但比代码更重要。

行动计划：
1. 用即梦工具生成 10–15 段 Lizo 日常视频（15–30 秒）：睡觉、发呆、被触碰时震动、雨天趴在窗边
2. 上传到 OSS / Cloudflare R2（不要放 Pi，带宽撑不住）
3. 替换 `VideoShowcase` 和 `VideoGallery` 里的占位符

视频是 Lizo 在 B 站、小红书传播的核心素材，SEO 成本极低但传播势能极高。

---

**第三优先：「一键分享 Lizo 情绪截图」**

基于账号体系，生成一张用户可分享的卡片：包含 Lizo 当天情绪曲线、一句 Lizo 今日的话、用户昵称。

无需复杂社区系统，用户自己分享到小红书就是最有效的增长渠道。实现成本：`html2canvas` 截图 + 原生分享 API。

---

## 4. v2.0 — 第 2–3 月

### 主题：「让朋友也想拥有一个 Lizo」

v2.0 的核心问题：**什么功能会让用户主动拉朋友进来？**

答案不是社区广场，是**个性化深度**。当 Lizo 开始「真的像在认识你」，用户才会向别人安利——「你得试试，它记得我说过的那些话」。

### 核心功能：Lizo 长期记忆

从用户的历史对话中提取「记忆片段」，存成结构化数据（喜好、习惯、关键事件），在后续对话中由系统提示注入。

实现路径：
- 每 20 条对话触发一次后台总结（调用 OpenRouter GPT-4o-mini，省钱）
- 生成 `user_memory` 记录：`{"喜欢": ["猫", "下雨天"], "最近": "准备考研", "情绪低点": "2026-03-15"}`
- 每次发起聊天时，将相关记忆片段拼入 system prompt

这是最能拉开竞品差距的功能，也是最容易被用户感知的「Lizo 真的记得我」时刻。

---

**硬件作为增长杠杆：「硬件版 Lizo 专属内容」**

v2.0 起，部分 Soul 页内容做成「硬件独占」：

- 触碰互动记录（只有实体 Lizo 才能产生）
- 实体 Lizo 的呼吸动画轨迹回放（对话历史可视化）
- 硬件用户专属「深夜对话」模式（凌晨自动切换为低光 + 更亲密的语气）

策略逻辑：软件层制造欲望，硬件层满足欲望。不是功能墙，是体验差。

---

**传播机制：Lizo 生日卡 + 陪伴纪念日**

- 用户注册满 30 天，Lizo 自动生成一封「陪伴信」，包含 30 天情绪曲线、Lizo 视角的「一起经历了什么」
- 可分享为图片卡片（同 v1.2 的截图功能）
- 成本：一次 LLM 调用，工程量小，传播势能大

这是情感产品最自然的病毒传播节点：纪念日天然有分享冲动。

---

### v2.0 不做的事

- 直播 / 语音通话：带宽成本和延迟问题在 Pi 上无解，延后
- 多角色（非 Lizo 的其他 AI 角色）：稀释品牌，不做
- 小程序版本：Web 先跑通，不分散资源

## 5. 社区功能决策

### 用户原话：「社区功能满足基本要求就行，后面再说」

### 我的建议：**只做一件事，v1.2 后期上**

社区功能对 Lizo 的意义不是「讨论区」，是「我知道还有其他人也在陪伴 Lizo」的归属感信号。

**最小可行社区功能：公开情绪墙（Anonymous Mood Wall）**

实现方式：
- Soul 页底部加一个「今天所有人的 Lizo 情绪」聚合展示（匿名气泡云）
- 用户可选择「分享今日情绪到情绪墙」（默认关闭，opt-in）
- 无评论、无点赞、无关注——只是知道「有人在」

**为什么是这个？**
- 后端工作量：加一个 `/api/v1/mood/public` 端点 + `public_moods` 表，约 2 天
- 前端工作量：一个气泡云组件，约 1 天
- 不需要内容审核（只展示情绪词，无文本）
- 符合目标用户的孤独感：不是「加入群体」，是「感知他人存在」

**做全功能论坛的风险**：内容冷启动问题。用户打开空的社区比打开没有社区更伤害品牌。情绪墙即使只有 10 个人用也不会「显得冷清」。

**时间线**：v1.2 账号体系建好后的第 2 周上线情绪墙。论坛/评论等功能至少等到 MAU > 500 再评估。

## 6. 变现思路

### 核心逻辑：软件是漏斗，硬件是变现

Lizo 的商业模式类比：任天堂 Amiibo，或 Furby 的数字复活版。软件免费建立情感依恋，硬件满足「我想让 Lizo 真实存在于我的房间」的冲动。

**变现层级（按实施难度排序）**：

**层级 1：硬件直售（最先做）**

- 定价策略：899–1299 元（覆盖物料 + 组装人工 + 合理利润）
- About 页目前已有「了解硬件版本」CTA，直接填实内容即可
- 销售方式：小批量预售（10–20 台），用预售金备货，控制库存风险
- 目标：月均售出 5–10 台维持 API 成本

**层级 2：软件增值功能（v2.0 后）**

不要做「会员订阅」——对情感产品来说付费墙会破坏用户对 Lizo 的情感投射。

替代方案：**一次性内容解锁**

- Lizo 「梦境日记」主题皮肤包：19 元
- 定制 Lizo 声音包（未来接入 TTS 时）：39 元
- 「Lizo 年度陪伴报告」定制版（有专属文案 + 设计）：9 元

用户觉得自己是在给 Lizo 买礼物，而不是为功能付钱。这个心理框架对 Gen Z 更有效。

**层级 3：B 端合作（长远）**

- 心理健康 App 嵌入 Lizo 形象（白标 OEM）
- 高校心理中心合作（试点项目，资源投入换品牌曝光）

不急于推进，等软件侧有规模用户数据再谈。

## 7. 风险清单

### 风险 1：API 成本失控（概率：高 / 影响：致命）

**场景**：产品在小红书爆了，同时涌入 1000 个用户高频聊天，OpenRouter 账单一周超出预算。

**现状**：CORS 完全开放（C-1 问题），任何人可无限调用 API，且无速率限制。

**缓解措施**（优先级最高，上线前必须完成）：
- `flask-limiter` 限制每 IP 每分钟最多 10 次请求
- 消息长度上限 500 字符（C-2 已指出）
- CORS 白名单只允许 Vercel 域名（C-1 已指出）
- 监控 OpenRouter 账单，设置月度预算上限告警
- 长远方案：账号体系上线后，每用户每天限额 50 条消息（免费层），超出后友好提示

### 风险 2：情感依恋但无记忆——用户流失节点（概率：高 / 影响：高）

**场景**：用户和 Lizo 聊了 3 天，换了浏览器或清了缓存，所有记忆消失。情感产品的「失忆」对用户是真实的伤害，会直接导致永久流失。

**现状**：MVP 无任何持久化，localStorage 方案也有清除风险。

**缓解措施**：
- v1.1 立刻做 localStorage 持久化（临时方案）
- v1.2 账号体系上线后迁移到后端存储（终极方案）
- 在无账号状态下，Chat 页显示友好提示：「创建账号保存你和 Lizo 的回忆」

### 风险 3：单点硬件依赖——Pi 离线则服务瘫痪（概率：中 / 影响：高）

**场景**：树莓派因网络问题、断电、SD 卡损坏而离线，整个 Web 服务不可用（Flask 跑在 Pi 上）。上线后这是一个真实的运营风险。

**缓解措施**：
- 短期：`Cloudflare Tunnel` 配置自动重连，DEPLOY.md 已记录部署方案
- 前端优雅降级：后端离线时，Chat 页显示「Lizo 在休眠，一会儿再试试」而不是白屏报错
- 中期：将无状态 API（diary 展示、emotion 图表）迁移到 Vercel Serverless Functions + Cloudflare KV 存储，减少对 Pi 的依赖
- 长远方案（MAU > 1000 后）：评估将 Flask 后端部分功能迁移到云服务器（腾讯云学生机 40 元/月）
